## 1. 目标

为当前这套第三方业务集成栈补一个独立的统一入口页，让使用者可以：

- 直接进入各业务页
- 在首页查看每个业务的当前状态
- 执行常用启停控制
- 快速看出问题是在容器、入口还是认证链路

首版按本地自用场景设计，不增加登录保护。

## 2. 交付物

首版设计对应的实现交付物应包含：

- 一个独立运行的轻量 Web 控制台服务
- 一个统一业务首页
- 业务状态探测接口
- 业务控制接口（`start` / `stop` / `restart`）
- 业务定义与映射配置
- 对应测试与验收记录

## 3. 非目标

首版不包含以下内容：

- 接入统一登录保护
- 日志中心、审计中心或历史趋势面板
- 任意命令执行能力
- 自动修复异常链路
- 深度端到端登录回放

## 4. 设计结论

采用 `轻量控制台` 方案：

- 新增一个小型 Web 服务，作为整套业务的统一首页
- 不挂进现有 Keycloak、Portainer、KafkaUI 或 Harbor 内部
- 页面展示业务总览、卡片列表、状态标签与常用控制按钮
- 后端复用现有 `compose.yml`、`.env`、`scripts/services.sh` 和 Harbor installer 结构

不采用纯静态首页的原因：

- 无法稳定承载实时状态探测
- 不适合做受控的启停动作

不采用监控型控制台的原因：

- 首版范围会迅速膨胀
- 当前项目没有必要一步做成完整平台

## 5. 页面结构

首页结构固定为两层：

### 5.1 顶部总览区

展示当前全局信息：

- 业务总数
- 正常数量
- 部分异常数量
- 异常数量
- 未安装数量
- 全栈控制按钮：`start`、`stop`、`restart`
- 全局最后刷新时间

### 5.2 业务卡片区

每张卡片对应一个“业务单元”，卡片内展示：

- 业务名称
- 业务描述
- 入口链接
- 主状态：`正常` / `部分异常` / `异常` / `未安装`
- 次级状态：`容器`、`入口`、`认证`
- 失败原因摘要
- 操作按钮：`打开`、`start`、`stop`、`restart`

必要时支持展开详情，查看最近一次探测结果。

## 6. 业务单元定义

控制台不直接把单个容器暴露给页面，而是使用预定义业务单元。

建议首版定义如下：

- `keycloak`
  - 包含：`keycloak`、`keycloak-postgres`
- `portainer`
  - 包含：`portainer`
- `kafka_ui`
  - 包含：`kafka-ui`、`kafka`
- `redisinsight`
  - 包含：`redisinsight`、`redis`、`oauth2-proxy-redisinsight`
- `phpmyadmin`
  - 包含：`mariadb`、`phpmyadmin`、`oauth2-proxy-phpmyadmin`
- `mongo_express`
  - 包含：`mongodb`、`mongo-express`、`oauth2-proxy-mongo-express`
- `harbor`
  - 来源：`harbor/installer/` 下 Harbor 官方生成 compose

这样做的目的：

- 页面语言与用户理解中的“业务页”一致
- 避免把页面做成底层容器控制器
- 为共享依赖保护预留边界

## 7. 状态模型

每个业务单元由三层状态组成，最终汇总成主状态。

### 7.1 第一层：容器状态

数据来源：

- 主栈：`docker compose --env-file .env -f compose.yml ps`
- Harbor：`harbor/installer/` 下的 `docker compose ps`

目标：

- 判断容器是否启动
- 判断是否存在明显退出、缺失或未安装

### 7.2 第二层：入口状态

数据来源：

- 面板服务主动探测业务 URL

目标：

- 判断业务入口是否可达
- 判断是否超时、连接失败或返回明显异常

示例：

- Keycloak：检查 realm 相关 OIDC 元数据或已知入口
- Harbor：优先检查 `api/v2.0/ping`
- 其他 Web 业务：检查首页或已知健康入口

### 7.3 第三层：认证状态

认证检查按业务接入方式分别处理，不使用统一探针硬套所有服务。

业务分组如下：

- `KafkaUI`
  - 当前通过原生 OIDC 接入 Keycloak
  - 未登录访问时，应出现到 Keycloak 授权地址的预期跳转
- `RedisInsight`、`phpMyAdmin`、`mongo-express`
  - 当前通过 `oauth2-proxy` 前置保护
  - 未登录访问时，应先进入 `oauth2-proxy` 流程，再导向 Keycloak
- `Keycloak`
  - 作为认证中心
  - 应能正确提供 realm 与 OIDC 元数据
- `Harbor`
  - 首版拆分为“服务可达”和“OIDC 配置检查”两项
  - 若 Harbor 未安装，则标记为 `未安装`
- `Portainer`
  - 首版仅做“容器 + 入口”检查
  - 不宣称具备稳定端到端认证检查，直到仓库里补齐可验证的 Portainer SSO 配置依据

### 7.4 主状态汇总规则

- `正常`
  - 容器、入口、认证均正常
- `部分异常`
  - 至少一个子状态失败，但业务仍可部分访问
- `异常`
  - 容器未运行，或入口不可达，或关键认证链路异常
- `未安装`
  - Harbor installer 缺失，或业务定义明确要求外部安装但当前不存在

## 8. 控制模型

首版同时支持“全栈控制”和“业务单元控制”。

### 8.1 全栈控制

顶部全局按钮直接复用现有脚本：

- `start` -> `./scripts/services.sh start`
- `stop` -> `./scripts/services.sh stop`
- `restart` -> `./scripts/services.sh restart`

### 8.2 业务单元控制

卡片按钮只允许控制预定义业务单元，不允许传入任意命令。

执行方式：

- 主栈业务通过固定参数调用 `docker compose`
- Harbor 业务在 `harbor/installer/` 下调用 Harbor compose

动作白名单仅允许：

- `start`
- `stop`
- `restart`

### 8.3 共享依赖保护

首版必须避免以下问题：

- 因控制某张业务卡片而意外停掉共享基础设施
- 两个控制动作并发执行导致状态错乱
- 页面直接拼接 shell 命令

约束如下：

- 页面只上传 `unit_id + action`
- 后端维护固定白名单映射
- 同一时间仅允许一个控制任务执行
- 对共享依赖采取保守策略
  - 若业务单元依赖共享基础设施，首版优先允许 `start`
  - 对可能影响其他业务的 `stop` / `restart`，需明确标出影响范围并按映射执行

## 9. 首版技术方案

建议使用：

- 后端：`Python`
- 前端：纯静态 `HTML + CSS + JavaScript`

原因：

- 当前项目没有现成的前端工程
- 当前环境已有 Python 可直接运行
- 首版需求以面板、接口和状态控制为主，不需要引入重前端构建链
- Python 更容易安全地包裹现有脚本与 compose 命令

服务职责：

- 读取 `.env` 与业务定义，生成入口信息
- 提供状态查询接口
- 提供控制接口
- 负责按需探测和刷新业务状态
- 返回最近一次探测与控制结果

## 10. 数据流

首页数据流如下：

1. 页面加载后请求业务总览接口
2. 后端读取最新缓存或触发一次状态刷新
3. 后端分别收集容器状态、入口状态、认证状态
4. 后端汇总主状态并返回卡片数据
5. 前端渲染卡片与总览
6. 用户点击控制按钮时，前端发送 `unit_id + action`
7. 后端执行白名单动作并返回执行结果
8. 前端收到结果后刷新状态

## 11. 错误处理

首版错误处理以“可见、可定位、不中断其他卡片”为原则。

处理策略：

- 单个业务探测失败，不影响其他业务状态返回
- 卡片直接显示最近一次失败摘要
- 控制动作失败时，前端保留错误原因，不做假成功
- 超时、连接失败、compose 执行失败、Harbor 未安装都应显示为明确文案
- 前端按钮在控制执行中短暂禁用，避免重复点击

## 12. 测试与验收

### 12.1 自动化测试

至少覆盖：

- 业务单元映射解析
- 状态汇总规则
- 动作白名单校验
- 控制接口拒绝非法参数

### 12.2 集成验证

至少验证：

- 首页能展示全部业务卡片
- 链接地址来自实际配置
- `start` / `stop` / `restart` 能触发预期动作
- 业务异常时能在页面上看到对应失败点
- Harbor 缺失时不会把状态误报为正常

### 12.3 人工验收

满足以下条件时可认为首版可用：

1. 能从统一首页直达各业务页
2. 能区分容器异常、入口异常、认证异常
3. 能执行全栈启停
4. 能执行业务单元级启停
5. 失败时页面上能看出具体问题点

## 13. 风险与边界

- `Portainer` 当前缺少可直接验证的完整 SSO 配置依据，因此首版不承诺其认证链路深检
- `Harbor` 采用官方 installer 路线，状态采集和控制需兼容其独立 compose
- 若 `.env` 缺失或配置不完整，面板必须显式报错，而不是展示伪状态
- 当前目录不是 git 仓库，无法满足“写完设计文档后提交”这一流程要求，本轮仅能保存文档文件
- 当前项目没有 `.gitignore`，若后续纳入 git，建议忽略 `.superpowers/`

## 14. 冻结决策

本设计确认后的冻结结论如下：

- 采用独立轻量控制台，不挂进现有业务内部
- 首版不加登录保护，按本地自用场景设计
- 首页同时提供业务直达、状态展示和启停控制
- 状态模型固定为“容器 + 入口 + 认证”三层
- 控制动作固定为白名单，不开放任意命令执行
- 首版技术路线固定为 `Python + 静态 HTML/JS`
